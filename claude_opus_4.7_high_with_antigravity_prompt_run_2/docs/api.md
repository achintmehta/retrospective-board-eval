# API reference

The backend exposes a REST API for CRUD operations and a Socket.io channel
for real-time collaboration on a board. Both share the same Express server
(`server.js`).

- **Base URL (dev):** `http://localhost:4000`
- **Base URL (prod / Docker):** `http://localhost:4000`
- **WebSocket path:** `/socket.io` (default)

## Conventions

- All request and response bodies are JSON unless explicitly noted.
- `created_at` and `position` are numeric (ms since epoch, float positions).
- IDs are 12-character alphanumeric strings generated by `nanoid`.
- Errors return `{ "error": "<message>" }` with the appropriate status code.

## REST endpoints

### `GET /api/health`

Liveness check. Returns `{ ok: true, uptime: <seconds> }`.

### `GET /api/boards`

List every board, newest first. Each entry includes denormalized counts.

```json
[
  {
    "id": "abC123XYZdef",
    "title": "Sprint 42 retro",
    "created_at": 1719158400000,
    "column_count": 3,
    "card_count": 7
  }
]
```

### `POST /api/boards`

Create a new board. Body: `{ "title": "Sprint 42 retro" }`.

- Three default columns (`Went Well`, `To Improve`, `Action Items`) are
  created automatically.
- `400` if `title` is missing or empty.
- `201` returns `{ id, title, created_at }`.

### `GET /api/boards/:id`

Returns the full board state: board metadata, ordered columns, ordered
cards per column, and ordered comments per card.

```json
{
  "id": "abC123XYZdef",
  "title": "Sprint 42 retro",
  "created_at": 1719158400000,
  "columns": [
    {
      "id": "col1",
      "board_id": "abC123XYZdef",
      "title": "Went Well",
      "position": 1000,
      "created_at": 1719158400000,
      "cards": [
        {
          "id": "card1",
          "column_id": "col1",
          "content": "Shipped the new dashboard",
          "author_name": "Alex",
          "position": 1000,
          "created_at": 1719158400500,
          "comments": [
            {
              "id": "cmt1",
              "card_id": "card1",
              "content": "Loved the new charts!",
              "author_name": "Sam",
              "created_at": 1719158460000
            }
          ]
        }
      ]
    }
  ]
}
```

`404` when the board does not exist.

### `POST /api/boards/:id/columns`

Add a new column to a board. Body: `{ "title": "Kudos" }`.

- `400` if `title` is missing or empty.
- `404` if the board does not exist.
- `201` returns the column `{ id, board_id, title, position, created_at }`.

### `GET /api/boards/:id/export`

Streams a CSV file of the entire board. Response headers:

```
Content-Type: text/csv; charset=utf-8
Content-Disposition: attachment; filename="<slug>-<board-id>.csv"
```

Each row represents either a card (`row_type=card`), a comment
(`row_type=comment`, denormalised against its parent card), or an empty
column (`row_type=column-empty`). Columns:

```
board_id, board_title,
column_id, column_title,
row_type,
card_id, card_content, card_author, card_created_at,
comment_id, comment_content, comment_author, comment_created_at
```

## Socket.io events

The frontend opens a single Socket.io connection per browser tab and joins
the room for the board it is viewing. Rooms are scoped per board, so an
event only reaches the clients viewing that board.

All emit-from-client events accept an optional **ack** callback that
receives `{ ok: true, ... }` on success or `{ ok: false, error }` on
failure.

### Client → Server

| Event         | Payload                                                 | Effect                                                                  |
| ------------- | ------------------------------------------------------- | ----------------------------------------------------------------------- |
| `join_board`  | `{ boardId, displayName }`                              | Socket joins the `board:<boardId>` room. Triggers a `presence_update`.  |
| `leave_board` | `{ boardId }`                                           | Socket leaves the room. Triggers a `presence_update` to remaining peers.|
| `add_card`    | `{ columnId, content, authorName }`                     | Persists the card, broadcasts `card_added` to the board's room.         |
| `move_card`   | `{ cardId, toColumnId, toIndex }`                       | Updates the card's column/position, broadcasts `card_moved`.            |
| `add_comment` | `{ cardId, content, authorName }`                       | Persists the comment, broadcasts `comment_added`.                       |

### Server → Client

| Event             | Payload                                                       | Notes                                                                   |
| ----------------- | ------------------------------------------------------------- | ----------------------------------------------------------------------- |
| `presence_update` | `{ boardId, peerCount }`                                      | Sent on join, leave, and disconnect.                                    |
| `peer_joined`     | `{ boardId, displayName }`                                    | Sent to *other* clients when a new user joins.                          |
| `card_added`      | `{ card }`                                                    | `card` includes `column_id`, `position`, `comments: []`.                |
| `card_moved`      | `{ cardId, toColumnId, toIndex, position }`                   | Clients should remove the card from its old column and reinsert.        |
| `comment_added`   | `{ comment }`                                                 | `comment.card_id` identifies the parent card.                           |

## Data model

```text
boards          (id PK, title, created_at)
board_columns   (id PK, board_id FK, title, position, created_at)
cards           (id PK, column_id FK, content, author_name, position, created_at)
comments        (id PK, card_id FK, content, author_name, created_at)
```

- `ON DELETE CASCADE` is enabled on every foreign key.
- SQLite is opened with `journal_mode=WAL` for concurrent reads alongside
  writes.
- `position` is a real number; on `move_card`, the destination column's
  cards are repacked to evenly-spaced integers (`1000, 2000, …`) to keep
  ordering stable under heavy churn.

## Error handling

- REST validation errors return `400` with a short `error` message.
- Missing resources return `404`.
- Socket.io handlers return `{ ok: false, error }` via the ack callback;
  invalid payloads do not produce a broadcast.

## Limits

- Card / comment content is truncated to 1000 characters server-side.
- Author and column titles are truncated to 40 / 80 characters server-side.
- JSON request bodies are capped at 256 KB.
