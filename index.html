Architecture Quality

Implementation Quality Rubric

Inspired by the Playwright vs. no-Playwright case study, this rubric measures structural quality independently from functional score. Six dimensions, each scored 1-3 (18 max). A model can ship a fully working app (45/45) while scoring poorly here. The two dimensions are orthogonal, and that's kind of the point.

Component Decomposition
3 5+ React component files  ·  2 2-4 components  ·  1 Monolithic pages
CSS Architecture
3 5+ files or 500+ lines or design token system  ·  2 100-499 lines  ·  1 Under 100 lines or inline styles
Backend Error Handling
3 5+ try-catch blocks in routes  ·  2 2-4 try-catch blocks  ·  1 0-1 try-catch blocks
Frontend Robustness
3 Dedicated API client module  ·  2 Inline response.ok checks  ·  1 No error checking on fetch calls
Code Modularity
3 Route files + db module (backend) AND API client (frontend)  ·  2 One axis only  ·  1 Monolithic
DnD Robustness
3 Position-sorted, reliable on first try  ·  2 Works or fixed after prompting  ·  1 Permanently broken
The Core Paradox

The same model (Opus 4.6 xHigh) scores 14/18 AQ without Playwright and only 6/18 with it. That's the lowest score in the dataset. Playwright enabled a "write fast, verify visually, move on" strategy that bypassed the defensive architecture the model would otherwise have built. The numbers: 23 CSS lines (inline) vs. 467 lines across 8 files; zero server try-catch blocks vs. partial error handling; a DnD implementation that crashes the app vs. one with position-sorted robust state. The verification shortcut cost 8 architecture points.

Key Findings

What the Data Shows

Finding 01
Model Generation

Opus 4.7 is the only model where every config scored 45/45

All five Opus 4.7 configurations (High, xHigh, with/without Playwright, with Antigravity prompt) shipped fully working applications without manual fixes. No other base model hit this consistency. It's a clear step change from 4.6.

Finding 02
Systemic Bug

Docker broke in 11 of 22 runs. Two bugs, both predictable

First: better-sqlite3 requires Python build tools absent in Alpine containers. Second: Express 5 (now the npm default) broke app.get('*') wildcard routing. Models that chose sql.js (pure WASM) and knew about Express 5 routing changes shipped working Docker containers every time.

Finding 03
Hardest Feature

Card drag-and-drop had the highest first-attempt failure rate

Moving cards between columns required WebSocket broadcasts, DB updates, and React state to all stay in sync. Any layer out of sync produces visible failures: blank dashboards after a move, ghost duplicate cards, or silent no-ops. Even top models needed a fix pass on this criterion.

Finding 04
Agent Design

Antigravity's browser sub-agent lifts Sonnet but can't compensate for weaker models

Antigravity (which spawns a GPT-OSS 120B browser sub-agent for visual QA) pushed Sonnet 4.6 to 45/45. But Antigravity Opus 4.6 still scored 43/45. Visual verification catches bugs, but it doesn't substitute for the model getting the code right in the first place.

Finding 05
Tooling

Playwright at xHigh is consistently excellent. At High, results vary

Playwright-enabled runs at xHigh were excellent across the board (Sonnet 4.6: 45/45, Opus 4.7: 45/45). At High effort, Playwright sometimes caught bugs it couldn't fully reproduce in the test environment. A Docker-specific SQLite type mismatch for Sonnet 4.6 High stayed unfixed at 42/45.

Finding 06
Effort Mode

xHigh effort helps weaker models more than strong ones

For Opus 4.7, both High and xHigh hit 45/45. The extra budget added nothing. For Opus 4.6, xHigh (43) clearly beat High (41). For Sonnet 4.6, xHigh (44-45) bested High (42). The stronger the base model, the less incremental value higher effort provides.

Finding 07
Non-Claude Models

Non-Claude models are mixed. Qwen Coder Next is the outlier

Gemini 3.1 Pro scored 43/45 (competitive with Opus 4.6). Qwen 3.6 got 40/45. Qwen Coder Next scored 27/45 at $179 orchestration cost, the worst result in the experiment. The local model simply couldn't follow complex multi-file agentic instructions reliably.

Finding 08
Best Value

Cost vs. quality is sharply non-linear

Sonnet 4.6 xHigh at $1.40 delivers 44/45. Sonnet 4.6 with Playwright at xHigh hits 45/45 for $2.57. Opus 4.7 High scores 45/45 at $3.15. The Qwen orchestration overhead ($41-$179) exceeded the cost of the best API runs by 13-57x while delivering worse results.

Finding 09
Library Choice

sql.js was the right SQLite choice. better-sqlite3 consistently broke Docker

The better-sqlite3 native module requires a Python toolchain and GLIBC version that isn't in Alpine or minimal Debian containers. Models that chose sql.js (a pure WASM port) shipped working Docker containers every time. A clear architectural signal for future benchmarks.

Finding 10
System Prompt

The Antigravity prompt improves aesthetics and code quality, but not functional scores for Opus 4.7

Adding Antigravity to Opus 4.7 produced more code (3,676 lines vs. ~2,200 baseline) and better docs. But it didn't improve on the already-perfect 45/45. It did lift Opus 4.6: xHigh plus Antigravity (44) edged ahead of base xHigh (43). The biggest impact is on visual aesthetics. The only 5-star UIs all came from antigravity-prompted runs.

Finding 11
Architecture

Playwright access degraded architecture quality by 8 AQ points in the Opus 4.6 xHigh pair

Opus 4.6 xHigh without Playwright scored 14/18 AQ. The same model with Playwright scored 6/18, the lowest in the dataset. Playwright enabled a verify-visually-and-move-on strategy that bypassed the defensive architecture the model would have built otherwise. Better tooling produced worse code.

Finding 12
Best Config

Opus 4.7 xHigh with Antigravity is the only config excellent across all three dimensions

Functional: 45/45. Architecture: 17/18 (highest in the dataset). Aesthetics: 5/5. No other configuration scored at the top tier on all three. It's also one of the more expensive runs at $5.59, but the gap in output quality is real.

Recommendations

What to Use

🏆 Best First-Shot Reliability

Claude Opus 4.7 at any effort level. Perfect 45/45 across all five tested configurations. Zero manual fixes required. Best choice when correctness on the first attempt is non-negotiable.

💰 Best Value (Cost + Quality)

Claude Sonnet 4.6 + Playwright at xHigh (45/45, $2.57). For a perfect run without Playwright: Opus 4.7 High at $3.15. Sonnet 4.6 xHigh alone ($1.40, 44/45) is the best near-perfect option if you want to keep costs low.

🏗 Best for Maintainability

Claude Opus 4.7 xHigh + Antigravity prompt: the only configuration that scores at the top across all three dimensions (45/45 functional, 17/18 architecture, 5/5 aesthetics). If you care about the code you'll inherit, not just the code that ships, this is the one.

🔁 For Iterative Workflows

Claude Sonnet 4.6 at xHigh (44/45, $1.40 after fixes) offers the lowest cost to near-perfect results. One fix needed for a native compile issue, easily addressed with a single follow-up prompt.

🤖 For Agent Pipelines

The Antigravity agent with Sonnet 4.6 delivers perfect functional results and exceptional aesthetics via combined coding and visual browser verification. An interesting multi-agent architecture worth replicating.

⚠️ Watch: Express 5 + better-sqlite3

Both are now npm defaults that reliably break Docker in agentic Node.js codegen. Prefer sql.js over better-sqlite3, and ensure agents know about Express 5's wildcard routing changes.

❌ Avoid: Qwen Coder Next for Full-Stack Tasks

27/45 with 11 criteria below perfect, worst result in the experiment. The Claude Opus 4.7 orchestration overhead hit $179 while core features stayed broken after 9+ hours. The local model couldn't reliably follow complex multi-file agentic instructions.
