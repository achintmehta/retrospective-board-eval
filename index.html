The Task

One Spec, Forty-One Runs

Every run got the same brief: build a self-hosted real-time retrospective board. React/Vite frontend, Node.js/Express backend, Socket.io for live sync, SQLite persistence. Must start with npm run dev and run in a single Docker container. Required features: board creation, configurable columns, guest display-name auth, drag-and-drop cards, nested comments, real-time broadcast, CSV export, and docs. Each run is scored on 14 functional criteria × 3 points = 42 max. Many configurations were run several times, so the dataset measures repeatability, not just a one-shot ranking.

How to read this page

Because the same configuration was run repeatedly, you will see multiple rows for, say, "Opus 4.6 · High". That is deliberate: the spread within a configuration is itself a finding. Treat a single row as one sample, not a verdict.

Variation & Unpredictability

The Same Prompt, Many Different Apps

A functional score hides how differently the agents actually built the app. Across the 41 runs — and even across repeats of one identical configuration — visual design, bonus features, code architecture, and documentation vary widely and largely independently. Any single agent output is one sample from a wide distribution, not a representative result.

Same configuration, different software

Re-running an identical setup does not converge on one app. For Opus 4.6 · High (no Playwright), seven runs of the exact same prompt produced functional scores from 38 to 42, container-restart persistence in 2 of 7, keyboard shortcuts in 1 of 7, READMEs from 53 to 116 lines, and 1 to 11 extra documentation files — no two runs sharing the same mix of extras.

Dimension 01
Visuals & aesthetics

From marketing-grade hero pages to default form styling

Visual output spanned the full 1–5 range: at one end, dark marketing-quality landing pages with hero copy, colour-coded columns and live badges; at the other, plain browser-default forms. It varies within a configuration too — the same Sonnet 4.6 + Playwright setup produced a bold indigo-header board in one run and a plain light-mode list in another. Theme (dark vs light), accent palette, and whether there is a landing page at all are effectively a per-run lottery.

Dimension 02
Unrequested features

Bonus features appear inconsistently, with no clear pattern

None of these were in the spec, yet across 41 runs: keyboard shortcuts (Cmd/Ctrl+Enter to submit) appeared in 17; a Docker VOLUME for restart persistence in 19; a live connected-users / presence indicator in only 4 (all Opus 4.7 + Antigravity prompt). Comment entry alone took four different shapes — a popup modal (e.g. Sonnet 4.6 High), a slide-in side drawer / panel (Antigravity Sonnet; Sonnet + Playwright), an inline expandable section on the card (Opus 4.6 runs), or inline directly on the board.

Dimension 03
Architecture

Code organization swings wildly, even at fixed settings

Source CSS ran from ~24 to over 1,100 lines; React components from 4 to 11; the backend from a single index.js to 6+ separated route/db modules; styling from all-inline to a full design-token system. Within Opus 4.6 · High alone the structure ranges from minimal-monolithic to richly decomposed (see the Architecture section) — same prompt, same model, very different codebases.

Dimension 04
Documentation

Docs range from a stub README to a full API reference

README length ran from ~50 to ~180 lines; dedicated documentation files from 0 to 19; some runs shipped complete backend and frontend API references, others none at all. Two runs of the same configuration could differ by an order of magnitude in documentation volume.

Tools & Capability

Does a Verification Tool Help?

The clearest controlled contrast in the dataset is Playwright (screenshot-based UI verification) on vs off, holding model and effort fixed. The anchor comparison is Opus 4.6 · High, with 7 runs on each side.

Stability
No effect

Playwright did not reduce bugs

Opus 4.6 (n=7 vs 7): non-Docker functional failures and severe bugs are statistically indistinguishable. The one severe drag-and-drop crash on each side is telling — a screenshot tool is passive and cannot click or drag, so it structurally can't exercise the interactive bugs that actually break this app.

Functionality
No effect

Scores are flat across the Playwright boundary

Opus 4.6: 40.6 → 40.6. Opus 4.7: 41.3 → 42 (n=1). Sonnet 4.6: 40.7 → 40.5. No Playwright run shipped a feature its matched non-Playwright peers lacked.

Cost
Not justified

Playwright costs more for nothing

Median session cost rises ~+27% on Opus 4.6 and ~+167% on Sonnet 4.6 with Playwright. Output token counts are equal, so the premium is verification overhead (screenshots, extra turns), not more code.

Tier vs Tool
Tier dominates

Capability matters ~10× more than tools

Pooled High-effort runs: Sonnet 4.6 averages 40.9/42 at $2.16, Opus 4.6/4.7 40.8/42 at $3.45, Qwen 30.5/42 at $41–179. The gap between tiers (~10 points) dwarfs anything a tool does within a tier (<1 point). And the cost penalty for a tool is proportionally worst on the cheaper model.

Scope: match the tool to the problem, not the other way around

This is a result about one tool on one problem — not a verdict on tooling in general, nor on Playwright specifically. Screenshot verification was a poor fit here because this app's failure modes are interactive (drag-and-drop, live multi-window sync) — exactly what a passive screenshot cannot exercise. A tool that can click, assert, drive the API, or run a real test suite — or a more visual or static-render problem — could easily repay its cost. The lesson is to configure tooling around a problem's actual failure surface; bolting tools on by default just drives up cost for no return.

Architecture & Variance

The "Playwright Shortcut" Was Run-to-Run Variance

An early single-run comparison suggested Playwright pushed the model toward monolithic, minimal-CSS, error-light code. Replication does not support a causal tool effect: the same architecture appears in non-Playwright runs, and good architecture appears with Playwright. Below: structural metrics measured directly from the code of Opus 4.6 · High runs, sorted by CSS volume.

What replication showed

CSS volume spans 24–627 lines with Playwright and no-Playwright runs interleaved across the entire range. run_7 (the original "shortcut" run, highlighted) is simply the low tail of a very wide distribution — its near-twins on minimal CSS and monolithic structure (high_run_4, high_run_6) used no Playwright. And the drag-and-drop crash struck both run_7 (Playwright) and high_run_1 (no Playwright, and the richest CSS in the set) — so the crash is independent of architecture investment. The with/without split explains essentially none of the variance.

Visual Aesthetics

Screenshot Review

A curated, identity-confirmed set rated 1–5 on direct screenshot review (visual polish, colour coherence, layout, professional feel). Not every replicate was re-rated; the gallery illustrates the driver of visual quality rather than scoring all 41 runs.

Key aesthetic finding

The system prompt is the single biggest design differentiator. Every 5-star UI used the Antigravity design prompt; without it, even Opus 4.7 at xHigh produces clean-but-plain interfaces (3/5). In the controlled Opus 4.6 comparison, Playwright made no aesthetic difference (both arms ~3.4/5), and effort mode and model generation barely moved visual quality. Aesthetics is driven by design instructions, not by tools, effort, or raw capability.

Notably, the design prompt carried no specific visual guidelines — no exact colour schemes, component styling, or layouts, just general direction (essentially "make it polished and professional"). That such a light, non-prescriptive nudge unlocks marketing-grade UIs suggests the ability to build a polished interface is already latent in the model and simply needs to be invoked explicitly — left to its defaults, the model produces clean-but-plain work.

Key Findings

What the Data Shows

Finding 01
Reproducibility

The same configuration varies a lot run to run

Identical Opus 4.6 · High setups produced scores from 38 to 42 and CSS volume from 24 to 627 lines. A single run is a sample, not a verdict — the central message of running each config repeatedly.

Finding 02
Model Generation

Opus 4.7 is a clear step up in reliability

Every Opus 4.7 run landed 41–42/42, and at xHigh all three runs were perfect (42/42). It also escaped the Docker trap that snared 4.6 (see Finding 04).

Finding 03
Tooling

Playwright added no stability, no functionality, and cost more

In the controlled Opus 4.6 comparison (n=7 vs 7) scores were identical (40.6 vs 40.6); a passive screenshot tool can't exercise the interactive bugs that matter. Median cost rose ~27% (Opus) to ~167% (Sonnet).

Finding 04
Systemic Bug

Docker is a near-universal trap — and it's capability-sensitive

Docker (criterion 2) failed in the majority of Claude Code runs, but the rate falls with capability: Opus 4.6 ~87%, Sonnet 4.6 ~44%, Opus 4.7 ~20%. Root cause is ecosystem, not logic: native better-sqlite3 won't build on Alpine. Stronger models escaped by switching to sql.js / node:sqlite.

Finding 05
Hardest Feature

Drag-and-drop and live sync are where bugs cluster

Moving cards (#7) and real-time updates (#9–11) are the most common functional failures across every tier. Basic CRUD is solved; event-driven collaborative state is not — and it's exactly what passive screenshot tools can't test.

Finding 06
Effort Mode

Effort buys reliability more dependably than tools do

High effort still produces occasional misses; Opus 4.7 at xHigh was perfect across all three runs and never failed Docker. Spending budget on effort moved quality; spending it on Playwright did not.

Finding 07
Non-Claude Models

Capability tier is the master variable

Frontier Claude (Sonnet 4.6, Opus 4.6/4.7) clusters at 38–42. Gemini 3.1 (via Antigravity) is competitive at 38–40. Qwen collapses: 37, and Coder Next at 24/42 with cascading unfixable bugs.

Finding 08
Best Value

Cost vs. quality is sharply non-linear

Sonnet 4.6 ties Opus on functional score at ~60% of the cost (40.9 vs 40.8). A clean Sonnet 4.6 · High run cost ~$1–1.80; Opus 4.7 · High hit 42/42 at $3.15. The Qwen orchestration overhead ($41–179) exceeded the best API runs by 13–57× for worse output.

Finding 09
Library Choice

sql.js beat better-sqlite3 for Docker

The native better-sqlite3 module needs a Python/build toolchain absent in Alpine. Runs that chose the pure-WASM sql.js (or Node's built-in node:sqlite) shipped working containers; it's the single clearest architectural signal for Docker success.

Finding 10
System Prompt

The design prompt is an aesthetic lever, not a functional one

Antigravity-prompted runs hold functional score flat (~41 across Sonnet and Opus) but produce every 5-star UI. Visual quality comes from design instructions, not from tools, effort, or model generation.

Finding 11
Orchestration

The expensive failure mode is strong harness + weak model

Claude Code driving Qwen Coder Next burned $179 — 57× a native Claude run — for the worst result, as the harness spent tokens compensating for a model that couldn't follow multi-file agentic instructions.

Finding 12
Architecture

Architecture quality is noisy and not tool-driven

Within fixed Opus 4.6 · High, CSS volume, component decomposition, and inline-styling vary independently and span the full range regardless of Playwright. The earlier "tool degraded architecture" result did not survive replication.

Recommendations

What to Use

🏆 Best first-shot reliability

Claude Opus 4.7 at any effort. Every run scored 41–42/42; xHigh was perfect across the board and never failed Docker. Choose it when first-attempt correctness matters most.

💰 Best value

Claude Sonnet 4.6 · High. It ties Opus on functional score at roughly 60% of the cost (~$1–1.80 per clean run). Step up to Opus 4.7 only when you need the reliability tail closed.

🔁 Spend on effort, not screenshots

If you have budget to improve a run, raise the effort mode before adding a verification tool. Effort moved quality (Opus 4.7 xHigh = perfect); Playwright did not, and cost 27–167% more.

🎨 For visual quality, use a design prompt

A strong design-oriented system prompt is the only thing that reliably lifted aesthetics. Tools, effort, and model generation barely moved it.

⚠️ Watch: better-sqlite3 + Alpine Docker

The most common single failure across all models. Prefer sql.js or node:sqlite, and verify drag-and-drop / live sync manually — the parts tools can't see.

❌ Avoid: strong harness driving a weak model

Qwen Coder Next via Claude Code scored 24/42 at $179 orchestration cost. Pairing a capable agent harness with a weak executor maximizes cost and minimizes quality.
